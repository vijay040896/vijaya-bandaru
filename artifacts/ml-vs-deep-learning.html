Machine Learning Approaches: Simple ML vs. Deep Learning

Introduction

For this class activity, I compared two real-world AI applications — one built on a simple, classic machine learning algorithm and one built on deep learning — to explain why each approach fits its problem, and why the other approach would not. Where my chatbot artifact shows hands-on AI agent building, this report shows a different skill: evaluating and reasoning about which AI/ML approach is the right tool for a given problem.

Description

I selected two examples: email spam detection, typically handled with algorithms like logistic regression or decision trees, and image-based facial recognition, handled with deep convolutional neural networks (CNNs). For each, I researched a real, named implementation rather than a hypothetical one, and analyzed the reasoning behind the algorithm choice.

Objective

My goal was to move past just naming algorithms and actually justify, with real-world evidence, why a given technique is the right (or wrong) fit for a specific kind of problem — the kind of judgment call that matters when deciding how to architect an AI solution, not just how to build one.

Process

Selecting the Examples

I picked one example representing traditional/simple ML (a classification problem with structured, engineered features) and one representing deep learning (a problem involving raw, high-dimensional, unstructured data), so the comparison would highlight a genuine contrast rather than two similar cases.

Grounding in Real Applications

Rather than relying on textbook descriptions, I researched actual production systems: Gmail's spam filter (which processes over 15 billion unwanted messages a day at roughly 99.9% accuracy using lightweight, feature-based classifiers) and Facebook's DeepFace system (launched in 2014, trained on four million images of four thousand people, reaching about 97.35% accuracy using a deep CNN) to make sure my reasoning was tied to verifiable, real-world evidence.

Analyzing the Fit

For spam detection, I reasoned that the relevant signals — word frequency, sender reputation, links, headers — are already well understood and easy to engineer, so a simple, interpretable, cheap-to-run model is the better fit at Gmail's massive scale.

For facial recognition, I reasoned the opposite: raw pixel data is too high-dimensional and variable (pose, lighting, angle) for hand-engineered features to capture reliably, which is exactly the problem a CNN is designed to solve by learning its own feature hierarchy directly from the image data.

Arguing the Counterfactual

To make the comparison rigorous, I also explained why the other approach would be a worse fit in each case — deep learning would be unnecessary computational overhead for spam detection, and simple ML would be structurally incapable of matching a CNN's accuracy on raw facial image data.

Tools and Technologies Used

Web research to verify real-world implementations, comparative technical analysis, and Microsoft Word for the final written report.

Value Proposition

Unique Value

This artifact shows a different, complementary skill to my chatbot project: not just building with AI, but critically evaluating which AI/ML technique actually fits a given problem, and being able to explain that reasoning clearly in writing.

Relevance

As I move toward building AI agents professionally, this kind of judgment — matching the right approach (a simple rule-based check vs. a full model, a lightweight classifier vs. a deep model) to the right problem — is exactly the skill that separates functional AI tools from well-architected ones.
